## 背景与问题归因

现状中，音频源文件实体与 UI 别名关系存储在 `audio_files.<sha256>.name.<name_id>`。

- `sha256`：真实文件内容身份（去重）
- `name_id`：同一 `sha256` 下的 UI 逻辑别名索引
- `sounds.*.source_file_for_sound`：引用 `sha256 + name_id + type`

问题的根因不是 `sha256` 去重本身，而是 `name_id` 被设计成“可复用槽位”而非“不可复用身份”。
当删除后重导入同一 `sha256` 时，历史缺失引用可能重新命中新的别名槽位，形成“无用户操作的自动复联”。

## 目标 / 非目标

### 目标
- 删除后缺失的源文件引用在未显式重新选择前保持缺失。
- 重导入相同文件不会复活历史引用。
- 依赖检查与实际播放行为一致，不出现“显示缺失但仍可播放”的隐式回连。

### 非目标
- 不改变 `sha256` 去重策略。
- 不重做 `audio_files` 主体结构（仍保留现有 `name` map 模型）。
- 不在本次变更引入全量历史配置离线迁移工具。

## 方案决策

### 决策 1：`name_id` 改为 UUID（字段不改名）
- 保留现有数据结构 `audio_files.<sha256>.name.<name_id>` 与引用结构 `source_file_for_sound.name_id`。
- 新导入同 `sha256` 时，`name_id` 不再分配为递增数字字符串，而是生成 UUID 字符串。
- 删除后重导入得到全新 UUID，因此历史引用不会被动命中新条目。

收益：
- 不需要引入额外 counter 元数据与生命周期管理。
- 保持接口字段不变，仅替换 ID 生成策略，改动集中且可控。

### 决策 2：播放前做严格三元引用校验
- 在声音播放解析时，将 `source_file_for_sound` 的 `sha256 + name_id + type` 与 `audio_files` 当前数据进行存在性校验。
- 若引用缺失，直接按“缺失依赖”处理，拒绝继续按 `sha256+type` 拼路径播放。

收益：
- 消除“UI 判定缺失但运行时仍可播放”的行为分裂。
- 与用户心智一致：未重新绑定，不应自动恢复。

## 备选方案与取舍

### 备选 A：递增不复用（counter）
- 优点：ID 具备顺序可读性。
- 缺点：需要新增并维护持久化游标，复杂度高于 UUID；仍要处理初始化推导与并发写入边界。

### 备选 B：引入全新 `source_ref_uuid` 字段并重构所有引用
- 优点：模型最干净。
- 缺点：改动面过大（前后端与配置迁移复杂），不符合本次“定点修复”目标。

### 备选 C：删除时墓碑标记
- 优点：无需改变 `name_id` 生成。
- 缺点：需要维护墓碑生命周期，且运行时/依赖校验一致性成本高。

结论：采用“`name_id` UUID 化 + 播放链路严格校验”作为当前最佳平衡。

## 兼容与迁移策略

- 旧配置：历史数字 `name_id` 保持可读、可删、可重命名。
- 新配置：后续新增条目改用 UUID `name_id`，允许数字与 UUID 在同一专辑并存。
- 不要求批量改写既有 `sounds`/`key_tone` 引用。

## 风险与缓解

- 风险：某些逻辑若隐含“`name_id` 为数字”的假设会失效。
  - 缓解：统一按字符串等值比较；在导入、重命名、删除、依赖校验、选项值拼接路径做专项回归。
- 风险：运行时严格校验后，部分历史“侥幸可播放”配置将按缺失处理。
  - 缓解：在依赖告警与编辑流程中提供明确的重新选择源文件路径。

## 验证策略

- 单元验证（Go）：
- 连续导入同 `sha256`，每次 `name_id` 均为新 UUID 且互不相同。
- 删除后重导入同 `sha256`，`name_id` 与历史已删除引用不相等。
- 集成验证：
  - 构造 `sounds` 引用被删除后，重导入同文件，不自动恢复依赖。
  - 仅在用户手动重新选择源文件并保存后，依赖恢复。
- 回归验证：
  - 多次导入相同文件仍只存储一份物理文件。
  - SSE 映射、列表展示、重命名与删除流程保持可用。
